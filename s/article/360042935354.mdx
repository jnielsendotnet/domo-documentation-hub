### Understand Your Users

When sharing and organizing data, think about the type of data consumers you have and the way they want to consume the data. For example, you may have some or all of the following:

* Consumers who will only interact with data on mobile.
* Consumers who need to receive scheduled reports.
* Consumers who are building cards and creating their own dashboards.
* Consumers who only need one DataSet—you can share a single card for them to build on.
* Groups that need to view only specific cards relevant to their divisions.

### Sharing Sensitive Data and PDP

* Depending on the size of your organization, a lead admin, executive stakeholders, and others (as needed) should determine organizational Personalized Data Permissions (PDP) policies.
    
  These policies should be communicated to everyone who will be sharing data so that it is kept secure.
* If your organization has DataSets with Personally Identifiable Information (PII), there should be specific PDP policies for sensitive data.
* Before sharing cards with anyone, create PDP policies to control data visibility in case those cards are shared beyond your immediate recipient(s).

### Establish Onboarding Permissions

When onboarding people to Domo, make sure you properly establish permissions for user access and abilities. This is done through system and custom roles. This list highlights some basic differences between the available system roles. See the [System Roles | Reference](/s/article/360043438953) and [Manage User Roles and Grants](/s/article/360043438973) to learn more.

* **Social users —** Access to Buzz conversations and shared cards.
* **Participant users —** View-only access to content (cards, dashboards). Participants can move and resize cards on a dashboard and share or export cards.
* **Editor users —** All participant abilities + edit access to content that they create or is shared with them, create and share content.
    
  **Note:** Editors *cannot* change permissions status.
* **Privileged users —** Ability to edit and access cards, delete cards, access data, delete user accounts, and assign system roles (except when inviting a new user).
* **Admin users** **—** All access right to content and user permissions.

### Use a Hub

We recommend that you create an individual hub—an implementation page, scratch pad, or whatever works for you—to house all the content you will be sharing. If you have a central point to go to for what you want to share, then you know which cards/dashboards/DataSets you are sharing and can easily access that content.

### Use Groups

Use groups to share content in Domo. By setting up specific groups, you can easily share the relevant data for that group and monitor who has access to the content that you’ve shared.

<Note>
  **Note:** When you add users to a default group, email notifications are sent to *all* group members.

</Note>

For more information about groups, see [Create and Manage User Groups](/s/article/360042934294 "Creating and Managing User Groups").
