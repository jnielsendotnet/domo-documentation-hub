### Intro

To get started with Domo Everywhere, you must establish a connection between the publishing instance and all of the subscribing instances. The Domo administrator for the publishing instance creates and maintains the connections to the subscribing instances from the Domo Everywhere page. See [Access Domo Everywhere](#access_domo_everywhere) below for more information on setting this up.

<Tip>
  **Definition:**
  The admin also configures the authentication for each user within the subscribing instances. This is called routing.

</Tip>

After configuring routing, the Domo admin must create each subscribing instance within the publishing instance and provide [key attributes](#define_key_attribute) for the individual users so that as they log in they are mapped to the appropriate subscribing instances. After the admin provides the key attributes, Domo completes mapping automatically. The admin can then view the mapping details from within the publishing instance.

### Subscriber Instance Setup

After setting up routing, the next step is adding subscriber organizations to receive your published content. A Domo admin can create, update, and delete any number of subscribing instances based on needs and workflows.

Before creating a subscriber instance for the first time, you must define your key attribute. Domo uses the key attribute to map a user to an instance. To help you determine your key attribute, consider which field in a user's Domo profile comes from your identity provider (IdP). Common examples are `customer ID` and `partner ID.` We recommend renaming the **Key attribute** field to your preferred value.

After you define your key attribute once, you can skip this section and begin with [Create a Subscriber Instance](#h_01GFC6JHXCVYJARCW15EPK2D5S).

### Mapping

After creating an instance, it's time to set its key attribute for mapping. Users are mapped according to the **Attribute Value** on the left side of the mapping table. As each user signs into the host portal, Domo redirects the iframe to the **Child instance URL** listed on the right side of the mapping table.

<Frame><img alt="mapping_table.png" src="/images/kb/ka0Vq0000000m6H-00N5w00000Ri7BU-0EM5w000005vXmn.png" style={{width: 609, height: 160}}/></Frame>
Follow these steps to map a subscriber instance:

1. From the navigation header, go to
**More** > **Admin**.

2. In the
**Domo Everywhere** menu, select **Embed**.

3. Go to the
**Mapping** tab.

4. Select
**+ Add Mapping**.

<Frame><img src="/images/kb/ka0Vq0000000m6H-00N5w00000Ri7BU-0EM5w000005vXmy.png" /></Frame>
The **Add mapping** dialog displays.

<Frame><img alt="add_mapping_dialog.png" src="/images/kb/ka0Vq0000000m6H-00N5w00000Ri7BU-0EM5w000005vXmq.png" style={{width: 530, height: 285}}/></Frame>
5. In the dialog, fill in the
**Attribute Value** and the **Child instance URL** for the user.

6. Select
**Save**.
  
The dialog closes.

Users are now mapped to the correct subscribing instance.

### Troubleshooting

**{'{"status":400,"statusReason":"Bad Request","message":"No value in attribute..."}'}**

This usually occurs if you have changed the Key Attribute name from the default and your host page has not adjusted to watch for the new name. The default always looks for "keyAttribute."

We recommend you change it to the field in your data and user profiles. Common examples include "Customer ID" and "Partner ID."

<Frame><img alt="ce6d562b-07ce-4ccc-bd99-6d5684843c34.png" src="/images/kb/ka0Vq0000000m6H-00N5w00000Ri7BU-0EM5w000005vXml.png" style={{width: 653, height: 156}}/></Frame>
**{'{"status":500,"statusReason":"Internal Server Error"...}'}**
  
This usually occurs if you or the automated process have not successfully uploaded the cert.pem to the SSO settings in the subscriber instance's Domo environment. Routing Certificates are usually uploaded automatically by the published configuration.

<Frame><img alt="cert_to_download_callout.png" src="/images/kb/ka0Vq0000000m6H-00N5w00000Ri7BU-0EM5w000005vXmm.png" style={{width: 795, height: 701}}/></Frame>
If that process was not successful, or if you are integrating a preexisting instance that was not created through a Domo Everywhere admin, you can also manually download the SSO certificate from the publishing organization and upload it to the subscriber instance.

For detailed instructions on how to do this, see [Downloading the Routing Certificate](#downloading-the-routing-certificate), above.

**"Browser cannot open page in embedded iframe"**
  
This usually happens if the **no-iframe-embedding** feature switch is active. Deactivation of the switch is normally done automatically as a part of the configuration published from the parent to the child along with the rest of the brand kit and subscriber feature switches.
  
If you click the "open site in new window" and the cert mentioned above has been uploaded, the child should load when accessed directly or when you click the "Open Site in New Window" button above.
  
If that published configuration was unsuccessful or the child is a preexisting instance, "no-iframe-embedding" can be activated by contacting your Domo representative. This will ensure the child environment also loads in embedded iframes.



### FAQ

<AccordionGroup>
<Accordion title="My customer has a certain template they apply to their users. How can I make sure that template is used?">

  
For this version, we require all subscribing instances to use the same "embedded end customer account" feature switch template. That template hides the Domo platform header and footer, removes Domo branding, and activates auto-subscription capabilities. However, if you have additional unique needs, you can coordinate additional template changes by contacting your CSM/AE.
  
Your custom template is then automatically used to create subscriber instances for that publisher.


</Accordion>
<Accordion title="I need to create multiple subscribers. Is there a bulk creation option?">

  
Bulk creation is not currently available.


</Accordion>
<Accordion title="Can I automate the bulk creation of instances programmatically via API?">

  
This is not currently available.


</Accordion>
<Accordion title="When a user logs into the customer portal, a new user is created in Domo and given an Admin default security role. Is it possible to change this setting, so that subscribers aren't Admins by default when no role is selected?">


Yes, you can set the default role at the instance level for each instance. It can be set to any standard or custom role.

</Accordion>
